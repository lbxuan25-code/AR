# New Surrogate / Inverse Repository Plan

## Purpose

Task J defines the future repository that will host dataset orchestration,
surrogate training, inverse search, and experiment fitting for AR spectra.

This document is a plan only. It does not create the new repository and does not
move training logic into the current forward repository.

## Repository Split

Current repository:

- role: forward-physics truth chain;
- owns normal-state, pairing, interface, BTK, RMFT-source projection, formal
  baseline, and AR-facing forward validation;
- exposes stable forward calls through `src/forward/` and
  `scripts/core/generate_forward_spectrum.py`;
- must not host large-scale training runs, checkpoint trees, inverse-search
  orchestration, or experiment-fitting output archives.

Future repository:

- role: training / inversion / experiment fitting workspace;
- owns dataset orchestration, surrogate training, model evaluation, inverse
  search, and experiment-fit reports;
- depends on this repository as a forward engine;
- must not copy normal-state, pairing, projection, interface, or BTK physics
  internals.

The boundary is intentionally sharp:

`forward truth repository -> stable forward interface -> training / inverse repository`

## Dependency Strategy

The new repository should consume this repository through one of two supported
paths:

- Python package import with `PYTHONPATH` / editable install and the public
  `forward` package;
- CLI calls to `scripts/core/generate_forward_spectrum.py` when process
  isolation is preferred.

Preferred programmatic imports:

```python
from forward import (
    BiasGrid,
    FitLayerSpectrumRequest,
    SourceRound2SpectrumRequest,
    TransportControls,
    generate_spectrum_from_fit_layer,
    generate_spectrum_from_source_round2,
)
```

Every generated dataset row must store the forward metadata fields emitted by
the forward interface, especially:

- `forward_interface_version`
- `output_schema_version`
- `pairing_convention_id`
- `formal_baseline_record`
- `formal_baseline_selection_rule`
- `projection_config`
- `git_commit`
- `git_dirty`

If any version or convention field changes, the training repository should
treat the resulting spectra as a new dataset family.

## Main Workflows

### Dataset Orchestration

The new repository should generate datasets by sampling Task-H fit-layer
controls and transport nuisance controls, then calling the current repository's
stable forward interface.

Dataset records should include:

- canonical forward request payload;
- canonical forward output payload or a compact reference to stored arrays;
- version metadata from the forward output;
- dataset split label;
- sampling policy identifier;
- any experimental preprocessing / normalization metadata.

The new repository should not reimplement `Delta(k)`, Fermi-surface extraction,
interface matching, or BTK conductance.

### Surrogate Training

Surrogate models should learn:

`fit-layer controls + transport controls -> AR spectrum / AR features`

They should not learn from raw RMFT source tensors unless a separate source-
linked dataset task explicitly defines that scope.

Training outputs should include:

- model checkpoints;
- training logs;
- validation metrics;
- calibration diagnostics;
- dataset version references;
- forward-interface metadata copied from the dataset.

### Evaluation

Evaluation should compare surrogate predictions against held-out forward
spectra generated by this repository's interface.

Recommended metrics:

- global spectral MSE;
- low-bias MSE;
- shoulder-region MSE;
- zero-bias conductance error;
- positive / negative peak bias error;
- peak-height error;
- uncertainty calibration when probabilistic models are used.

### Inverse Search

Inverse search should output candidate families, not a unique microscopic truth.

Each inverse result should include:

- ranked candidate family id;
- fit-layer pairing controls with accepted ranges;
- transport nuisance controls with accepted ranges;
- weak-channel branch status;
- objective / likelihood score;
- generated spectra and residual metrics;
- notes on degeneracy and non-identifiability.

The inverse repository may use surrogates for speed, but final reported
candidates should be rechecked with the current repository's forward interface.

### Experiment Fitting

Experiment-fitting outputs should keep experimental preprocessing separate from
physics claims.

Each report should store:

- raw experimental data reference;
- preprocessing / normalization policy;
- candidate-family results;
- forward-interface version metadata;
- final forward recheck spectra;
- limitations and degeneracies.
