# Order Parameter Refactor Round 2

## Goal

Round 2 replaces the restricted round-1 `PairingParams` ansatz with a
source-native physical channel layer while keeping the normal-state and BTK
transport workflow intact.

The current authoritative design is documented in
`docs/pairing_state_stage3.md`. This note summarizes the refactor boundary and
the current Stage-3 implementation state.

## Physical Channel Layer

The round-2 container is `PhysicalPairingChannels` in `src/core/parameters.py`.
It contains eight supported complex channels:

- `delta_zz_s`
- `delta_zz_d`
- `delta_xx_s`
- `delta_xx_d`
- `delta_zx_s`
- `delta_zx_d`
- `delta_perp_z`
- `delta_perp_x`

Stage 3 separates these into seven core truth-layer channels and one optional
weak channel:

- core channels:
  `delta_zz_s`, `delta_zz_d`, `delta_xx_s`, `delta_xx_d`,
  `delta_zx_d`, `delta_perp_z`, `delta_perp_x`
- optional weak channel:
  `delta_zx_s`

`delta_zx_s` remains available in code, but the default source projection
freezes it to zero unless it passes the Task-C soft gate.

## Source Projection

Round-2 source projection is implemented in `src/source/round2_projection.py`.
The current default path is not an unconstrained least-squares fit. It performs:

1. a single eV-to-meV conversion of Luo `delta_x`, `delta_y`, and `delta_z`,
2. a global gauge fix before fitting,
3. block weighting with `w_x = 1.0`, `w_y = 1.0`, and `w_z = 1.15`,
4. channel-wise ridge regularization,
5. a default soft-freeze decision for the optional `delta_zx_s` channel.

The projection remains physically interpretable: it reconstructs the source
tensors through explicit channel basis tensors rather than using an opaque
compression.

## Channel To Delta(k)

`src/core/pairing.py` supports both the legacy `PairingParams` container and
the round-2 `PhysicalPairingChannels` container.

The BTK path consumes the final pairing matrix `Delta(k)`. The round-2 scalar
terms are:

- `Delta_zz(k) = delta_zz_s * gamma_s(k) + delta_zz_d * gamma_d(k)`
- `Delta_xx(k) = delta_xx_s * gamma_s(k) + delta_xx_d * gamma_d(k)`
- `Delta_zx(k) = delta_zx_s * gamma_s(k) + delta_zx_d * gamma_d(k)`
- `Delta_perp_z(k) = delta_perp_z`
- `Delta_perp_x(k) = delta_perp_x`

These feed the existing `delta_matrix(k)` -> `interface_gap_diagnostics(...)`
-> `compute_multichannel_btk_conductance(...)` workflow.

## Formal Baseline

The formal round-2 baseline has a single authoritative source:

`outputs/source/round2_baseline_selection.json`

That record is generated by the source-side baseline builder using:

- Luo temperature-sweep RMFT source data,
- the charge-balanced `p ≈ 0` branch,
- `temperature_eV <= 1.0e-3`,
- the first 8 samples sorted by temperature,
- componentwise complex medians of the default-projected channels,
- the Task-C default weak-channel freeze convention, so `delta_zx_s = 0`.

Runtime code reads this record through `src/core/formal_baseline.py`.
`core.presets.base_physical_pairing_channels()` is therefore a loader for the
authoritative record, not an independent handwritten numerical baseline.

`compatibility_physical_pairing_channels()` is intentionally separate and only
serves legacy round-1-compatible comparison workflows.

## Metrics

Projection metrics are defined once in `src/source/projection_metrics.py`:

- `residual_norm_total = ||source - recon||_F`
- `retained_ratio_total = 1 - residual_norm_total / source_norm_total`
- `omitted_fraction_total = residual_norm_total / source_norm_total`

The same definitions are used by round-1 consistency diagnostics, round-2
projection diagnostics, and round-1-vs-round-2 comparison outputs.

Current source-side comparison summary:

- round-1 median retained ratio total: `0.34670871724588614`
- round-2 median retained ratio total: `0.36607361367472413`
- median retained-ratio improvement: `0.021674821882835316`
- round-1 median residual norm total: `59.27361303524721`
- round-2 median residual norm total: `54.63856967656513`

Round 2 is therefore a measurable improvement over round 1, but the improvement
is modest.

## Current Limitation

The round-2 truth layer is not full-RMFT-equivalent. Residual-anatomy diagnostics
show that most remaining residual weight lies outside the supported round-2
channel masks. The current layer should be described as a restricted,
physically interpretable projection of the RMFT source pairing tensor.

## Compatibility

The legacy `PairingParams` API still works through a compatibility conversion
in `src/core/pairing.py`. That compatibility path is preserved for tests and
explicit comparisons, but new round-2 forward work should use
`PhysicalPairingChannels` and the single authoritative formal baseline record.
